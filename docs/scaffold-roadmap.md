# 项目脚手架演进路线图

## 1. 目标

这份路线图不是做“功能堆砌”，而是把当前项目往“可复用的前后端分离管理后台脚手架”方向推进。

路线设计原则：

- 优先补平台型能力，不优先补单一业务页
- 优先补后续复用率高的模块
- 优先补能和现有能力形成闭环的模块
- 控制每一期范围，避免一次做太散

## 2. 当前项目定位

当前项目已经具备：

- 登录鉴权
- RBAC 菜单/按钮权限
- 用户/角色/菜单/配置/字典
- 在线用户
- 操作日志
- 消息推送中心雏形
- 下载中心与异步导出模板

所以你现在不适合再走“纯 CRUD 补页面”的路线，更适合开始沉淀基础设施模板。

## 3. 最值得做的 6 个模块

## 4. 第一优先级

### 4.1 登录日志

建议目标：

- 区分登录成功、失败、退出
- 记录用户名、IP、UA、结果、原因、时间
- 支持分页、筛选、导出

建议表：

- `sys_login_log`

建议页面：

- 系统管理 / 登录日志

建议接口：

- 登录日志分页查询
- 登录日志导出
- 最近登录信息查询

为什么先做：

- 成本低、收益高
- 和现有操作日志天然互补
- 可以顺手补安全审计能力

### 4.2 文件中心

建议目标：

- 统一管理头像、导出文件、后续附件
- 支持上传、列表、删除、下载、业务归类

建议表：

- `sys_file`

建议字段：

- `id`
- `biz_type`
- `biz_id`
- `storage_type`
- `origin_name`
- `stored_name`
- `file_ext`
- `content_type`
- `file_size`
- `file_path`
- `status`
- `creator`
- `create_time`

建议页面：

- 系统管理 / 文件中心

建议接口：

- 文件上传
- 文件分页
- 文件删除
- 文件下载地址获取

为什么先做：

- 你现在的头像和导出文件已经分散存在
- 后续导入、模板、附件都会依赖它

### 4.3 消息发送记录中心

建议目标：

- 给现有消息推送中心补完整闭环
- 能看到每次推送发给谁、走了哪些渠道、哪些成功失败

建议表：

- `sys_notify_record`
- `sys_notify_record_channel`

建议页面：

- 消息推送记录
- 消息详情页

建议接口：

- 发送记录分页
- 发送详情
- 失败重试

为什么先做：

- 你已经有渠道抽象了
- 差的不是“能发”，而是“可追踪、可复盘、可重试”

## 5. 第二优先级

### 5.1 定时任务中心

建议目标：

- 后台配置任务
- 手动执行
- 记录执行日志
- 为导出、消息重试、缓存维护提供调度基础

建议表：

- `sys_job`
- `sys_job_log`

建议页面：

- 系统管理 / 定时任务
- 任务执行日志

建议实现顺序：

1. 先做轻量内置版
2. 任务先限定为 Spring Bean + Method 调用
3. 后续复杂场景再评估 XXL-Job

### 5.2 通用导入中心

建议目标：

- 和你现在的下载中心组成“导入导出中心”
- 支持模板下载、异步导入、失败明细回执

建议表：

- `sys_import_record`
- `sys_import_record_error`

建议页面：

- 导入记录
- 导入错误明细

建议抽象：

- 参考你现在的异步导出模板
- 做一个 `AbstractAsyncImportService`

为什么这项值得做：

- 你已经把导出模板走通了
- 导入会非常自然，而且复用价值很高

### 5.3 组织架构 + 数据权限

建议目标：

- 从纯 RBAC 升级到“权限平台”
- 支持部门树、岗位、数据范围

建议表：

- `sys_dept`
- `sys_post`
- `sys_user_post`

建议扩展字段：

- `sys_user.dept_id`
- `sys_role.data_scope`

建议页面：

- 部门管理
- 岗位管理
- 角色数据权限配置

建议数据权限等级：

- 全部数据
- 本部门及以下
- 本部门
- 仅本人
- 自定义

为什么放第二优先级：

- 价值很高
- 但会改动用户、角色、查询模型，范围比前几项大

## 6. 第三优先级

### 6.1 代码生成器

建议目标：

- 用项目自己的规范生成 CRUD 模块
- 不是生成“通用 Java 文件”，而是生成“符合当前项目风格的模块”

建议覆盖：

- 后端 entity / mapper / service / controller
- 前端 api / 列表页 / 编辑弹窗
- 权限按钮
- 字典回显
- 导入导出挂钩

建议表：

- `gen_table`
- `gen_table_column`

为什么放第三优先级：

- 长期收益极高
- 但最好等组织、导入导出、文件中心这些基础模式再稳定一点

### 6.2 系统监控页

建议目标：

- 让后台具备基本运维排障能力

建议页面：

- 服务信息
- JVM 信息
- Redis 信息
- 缓存信息
- 磁盘/线程概览

适合时间点：

- 当项目开始稳定部署、多环境运行后再补

## 7. 推荐分期

### 第一期

- 登录日志
- 文件中心
- 消息发送记录中心

这一期完成后，你的项目会明显更像“可上线后台平台”。

### 第二期

- 定时任务中心
- 通用导入中心
- 下载中心升级为导入导出中心

这一期完成后，你的项目会明显更像“可复用脚手架”。

### 第三期

- 组织架构
- 数据权限
- 代码生成器

这一期完成后，项目会从后台模板升级为“平台底座”。

## 8. 如果只选 3 个最该先做的

如果你近期精力有限，我建议优先按这个顺序做：

1. 登录日志
2. 文件中心
3. 消息发送记录中心

原因很简单：

- 需求边界清晰
- 和现有系统结合最自然
- 不会一下子侵入太多已有代码
- 做完立刻能提升平台完整度

## 9. 一句话版本

你这个项目下一阶段最适合的方向，不是继续补零散页面，而是优先把下面三条线做完整：

- 审计线：登录日志
- 文件线：文件中心
- 消息线：发送记录与重试

再往后推进任务中心、导入中心、数据权限，整个项目就会越来越像一个成熟的开源脚手架底座。
