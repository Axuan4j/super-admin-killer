# 项目通用脚手架能力缺口分析

## 1. 分析范围

本文档基于当前仓库代码结构进行评估，参考的是常见开源前后端分离后台脚手架项目的通用能力模型，重点对比以下类型项目常见功能：

- Spring Boot + Vue 管理后台脚手架
- RBAC 权限管理类后台
- 运维监控与系统管理类中后台
- 带代码生成、审计、导入导出、定时任务的通用 admin 模板

当前分析对象：

- 后端：`sak-service`
- 前端：`sak-page`

## 3. 对标常见开源脚手架后，当前还比较缺的通用模板能力

下面这些不是“必须马上补”，但如果目标是往一个更完整、更像成熟开源 admin 脚手架的方向走，当前还缺得比较明显。

## 4. 高优先级缺口

这些功能对“平台通用性”和后续业务复用价值最高，建议优先补。

### 4.1 登录日志 / 安全审计日志

当前已有操作日志，但缺少一套独立的登录审计能力。

常见脚手架通常会单独提供：

- 登录成功日志
- 登录失败日志
- 退出登录日志
- 登录 IP、地域、浏览器、操作系统
- 异常登录告警基础能力

为什么建议补：

- 操作日志和登录日志关注点不同
- 安全排查时，登录轨迹通常比业务操作更重要
- 后续做账号锁定、风控、异地登录提醒会更顺

建议形态：

- `sys_login_log` 表
- 登录日志列表页
- 登录日志导出
- 最近登录信息展示

### 4.2 定时任务中心

目前没有发现 Quartz / XXL-Job / 内置任务调度管理模块。

常见脚手架通常会提供：

- 定时任务 CRUD
- Cron 表达式配置
- 手动执行一次
- 启停任务
- 执行日志

为什么建议补：

- 通知重试、数据归档、缓存刷新、报表生成都依赖任务调度
- 这是很多后台项目里复用率很高的通用基础设施

建议形态：

- 先做内置轻量版任务中心
- 后续如果任务复杂再接 XXL-Job

### 4.3 代码生成器

当前没有看到代码生成模块。

常见脚手架通常会有：

- 读取表结构
- 配置字段显示/校验/查询方式
- 一键生成 entity / mapper / service / controller
- 一键生成前端 API、列表页、表单页

为什么建议补：

- 你这个项目已经形成了相对稳定的后台开发模式
- 后续每加一个管理模块，重复代码很多
- 生成器能极大加速新模块接入

建议形态：

- 先做后端 CRUD 生成
- 再做前端列表页 + 表单页模板
- 最后接入“导出模板”“字典回显”“权限按钮”这些你项目自己的规范

### 4.4 文件中心 / 附件管理

目前有头像上传和导出文件下载，但还没有统一文件中心。

常见脚手架里通常会逐步演进出：

- 通用文件上传接口
- 文件列表
- 文件预览
- 文件归档/删除
- 按业务类型挂载附件

为什么建议补：

- 头像、导出、富文本附件、导入模板最终都会落到文件管理
- 没有统一文件中心，后面附件能力容易散在多个表和接口里

建议形态：

- `sys_file` 表
- 支持本地存储实现
- 后续可扩展 OSS / COS / MinIO

## 5. 中优先级缺口

这些能力不是立刻阻塞，但对“成熟后台平台感”帮助很大。

### 5.1 数据权限

当前权限模型主要是菜单/按钮级 RBAC，没有看到部门数据隔离、本人数据、角色数据范围等能力。

常见脚手架通常会支持：

- 全部数据
- 本部门数据
- 本部门及以下
- 仅本人
- 自定义数据权限

为什么建议补：

- 一旦进入真实业务，按钮权限远远不够
- 数据权限几乎是后台平台化的分水岭

### 5.2 部门 / 组织架构 / 岗位体系

目前用户和角色是平面的，没有看到组织树、部门、岗位模块。

常见脚手架一般会有：

- 部门树
- 岗位管理
- 用户归属部门
- 负责人、排序、状态

为什么建议补：

- 很多业务权限和消息分发都依赖组织结构
- 后续数据权限实现也经常依赖部门树

### 5.3 通用导入能力

当前导出能力已经不错，但“导入”这一半还没有。

常见脚手架通常会支持：

- Excel/CSV 导入
- 导入模板下载
- 导入校验错误报告
- 导入任务记录

为什么建议补：

- 用户、字典、配置、业务主数据的初始化很常见
- 你现在已有下载中心，补导入会很自然

建议形态：

- 与下载中心对应，逐步扩展成“导入导出中心”

### 5.4 系统监控页面

当前有在线用户和操作日志，但缺少更偏运维视角的监控页。

常见脚手架通常有：

- 服务信息
- JVM 信息
- 内存、线程、磁盘
- Redis 信息
- 缓存监控

为什么建议补：

- 后台项目一旦线上运行，排障经常需要这些基础信息
- 很适合放到“系统管理”或“监控中心”

### 5.5 消息中心的发送记录 / 回执 / 重试

你已经把消息中心抽象出来了，但还缺“平台化闭环”。

建议继续补：

- 消息发送记录表
- 单条消息各渠道发送状态
- 重试机制
- 失败原因
- 模板消息能力

这样“消息推送中心”才会从功能页升级成基础设施模块。

## 6. 低优先级但能提升模板完整度的能力

这些更偏“成熟度增强项”。

### 6.1 OpenAPI / Swagger 文档

当前没有看到 `springdoc` 或 Swagger UI 集成。

建议：

- 增加接口文档页
- 增加鉴权调试支持
- 后续让生成器、前端联调更顺

### 6.2 国际化 i18n

当前前端和后端文案看起来都是中文直写，适合当前阶段，但模板化程度还不够。

建议：

- 前端界面文案抽离
- 字典/状态枚举文案统一
- 如果后面想做 SaaS 或海外版本，会更从容

### 6.3 标签页导航 / 常用布局增强

很多开源后台会提供：

- 多标签页
- 面包屑
- 快捷搜索菜单
- 全屏
- 布局切换

这不是业务刚需，但会让管理后台更像成品模板。

### 6.4 通知公告 / 帮助中心 / 富文本内容管理

现在消息偏“推送”，但缺少“公告内容管理”。

可选增强：

- 公告管理
- 帮助文档管理
- 富文本内容页

### 6.5 前端工程规范能力

当前前端能构建，但没看到更完整的工程约束。

建议补：

- ESLint
- Prettier
- Stylelint
- 提交规范
- 基础组件封装规范

### 6.6 自动化测试体系

当前几乎没有形成完整测试模板。

建议补：

- 后端 service/controller 单测
- 前端组件/页面基础测试
- 核心流程冒烟测试

## 7. 如果按“通用后台平台”目标排序，建议优先补这 8 项

### P1

1. 登录日志
2. 定时任务中心
3. 文件中心
4. 代码生成器

### P2

5. 数据权限
6. 部门/岗位/组织树
7. 通用导入中心
8. 消息发送记录与重试

## 8. 结合你当前项目，最值得优先沉淀成模板的方向

你这个项目跟很多普通脚手架比，有一个很明显的优势：你已经开始形成“平台能力抽象”了，而不是只做静态 CRUD。

最值得继续强化的不是再堆几个普通表单页，而是下面 4 条主线：

### 8.1 消息中心平台化

从“发消息页面”升级为：

- 模板管理
- 渠道管理
- 发送记录
- 重试机制
- 用户订阅偏好

### 8.2 导入导出中心平台化

从“异步导出示例”升级为：

- 通用导出模板
- 通用导入模板
- 任务调度
- 文件归档
- 权限控制

### 8.3 RBAC 升级为“权限平台”

从“菜单按钮权限”升级为：

- 数据权限
- 组织架构
- 审计日志
- 缓存刷新与权限变更追踪

### 8.4 CRUD 升级为“代码生成规范”

把你现在已经稳定下来的模式沉淀为生成器模板：

- 列表页
- 编辑弹窗
- 按钮权限
- 字典项渲染
- 导入导出挂钩
- 操作日志埋点

## 9. 一句话结论

如果拿常见开源脚手架来对比，你这个项目现在“认证、RBAC、消息、异步导出”已经有不错基础，但还缺少成熟模板常见的 4 个关键平台模块：

- 登录审计
- 定时任务
- 代码生成
- 文件中心

如果先把这 4 个补上，再往数据权限、组织架构、导入中心推进，这个项目就会从“可用后台”明显升级成“可复用脚手架”。
